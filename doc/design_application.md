# アプリケーション層詳細設計書

本ドキュメントでは、Issue Manager アプリケーションにおけるアプリケーション層（ユースケース層）の責務、主要なユースケース、および外部とのインターフェースについて実装事実に基づいて記述します。

## 1. 概要
アプリケーション層は、ドメイン層のエンティティやサービスを組み合わせ、特定のユーザー操作（ユースケース）を実現する「調整役」としての責務を担います。ビジネスロジックそのものはドメイン層に任せ、入力のバリデーション（一部）、写真のアップロード、ドメインモデルの操作、および永続化の実行を制御します。

## 2. 責務
- **ユースケースの制御**: ユーザー操作に応じた一連の処理フロー（ワークフロー）を定義します。
- **依存関係の調整**: リポジトリやストレージといった外部サービスのインターフェースをドメイン層に注入し、具体的なインフラ実装からドメインを隔離します。
- **DTOの管理**: 外部（API/UI）とのデータのやり取りに使用するデータ構造（Command/Query）を定義します。
- **データ形式の変換**: ドメインエンティティとUI表示用データの相互変換（`toJSON()` 等の利用）を行います。

## 3. 主要なユースケース (Use Cases)

### 3.1 CreateIssueUseCase
新規の指摘事項を作成し、関連写真を保存します。

- **入力 (Command)**: タイトル、説明、カテゴリ、3D位置情報、作成者ID、ユーザーロール、階数、写真バイナリ、等。
- **処理フロー**:
    1. `PermissionPolicy.canCreate` による権限チェック。
    2. `IBlobStorageService` を使用して、受け取った写真バイナリを非同期でアップロード。
    3. `Issue.create` を使用して、アップロードされたURLを含むエンティティを生成。
    4. `IIssueRepository.save` によりDBへ永続化。
- **返却値**: 生成された指摘事項の UUID。

### 3.2 UpdateIssueUseCase
既存の指摘事項の属性（タイトル、説明等）を更新し、必要に応じて写真を追加します。

- **入力 (Command)**: 対象ID、更新内容（タイトル、説明、ステータス、カテゴリ、メタデータ）、更新者ID、ユーザーロール、追加写真バイナリ。
- **処理フロー**:
    1. リポジトリから既存エンティティを取得。
    2. `PermissionPolicy.canEdit` による権限チェック。
    3. 追加写真がある場合、`IBlobStorageService` でアップロードし、URLを取得。
    4. `issue.update` を呼び出し、属性とバージョン番号を更新。
    5. `IIssueRepository.save` による永続化。※リポジトリ層で楽観的ロック（versionチェック）が行われます。

### 3.3 ChangeIssueStatusUseCase
指摘事項のステータスのみを迅速に更新します。

- **入力 (Command)**: 対象ID、新ステータス、ユーザーロール、操作ユーザーID。
- **処理フロー**:
    1. リポジトリから既存エンティティを取得。
    2. `PermissionPolicy.canChangeStatus` による権限チェック。
    3. `issue.changeStatus` によるエンティティの状態遷移。※ドメイン層で「完了済みの再開」制限などがチェックされます。
    4. リポジトリに保存。

### 3.4 GetIssuesByFloorQuery
指定された階数に属する指摘事項を一覧取得します。

- **入力**: 階数（Floor ID）。
- **処理フロー**:
    1. `IIssueRepository.findByFloor` を呼び出し、エンティティの配列を取得。
    2. 各エンティティに対して `toJSON()` を実行し、プレーンなオブジェクト（`IssueProps`）の配列に変換。
- **返却値**: `IssueProps[]` （UI表示用データ）。

### 3.5 GetIssueByIdUseCase
IDを指定して単一の指摘事項を取得します。

- **入力**: 指摘事項 UUID。
- **返却値**: `Issue` エンティティ（または null）。

## 4. インターフェース (Interfaces)
アプリケーション層は以下の抽象インターフェースに依存しており、これによってインフラ実装の差し替えが可能になっています。

| インターフェース | 定義場所 | 責務 |
| :--- | :--- | :--- |
| `IIssueRepository` | `application/interfaces` | DBへの CRUD 操作、およびフロア検索。 |
| `IBlobStorageService` | `application/interfaces` | ファイル（写真）のアップロードおよび削除。 |

## 5. データ形式 (Data Structures)
- **Commands**: 変更操作に必要なデータをカプセル化したオブジェクト。
- **Plain Objects (DTO)**: ドメインエンティティから `toJSON()` によって生成され、UI層（Next.js API Route / Client Component）へ渡される非機能的なデータ。
